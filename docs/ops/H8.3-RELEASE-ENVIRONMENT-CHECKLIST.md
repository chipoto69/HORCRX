# Configure the `release` environment

The `.github/workflows/release.yml` workflow gates the `publish` job on a GitHub
environment named `release`. Until this environment exists in the repository
settings with required reviewers, the gate is yaml-only. This checklist closes
H8.3.

## Steps (operator action; one-time, ~3 minutes)

1. Go to `https://github.com/chipoto69/HORCRX/settings/environments`
2. Click "New environment"
3. Name: `release` (exact, lowercase)
4. Click "Configure environment"
5. Under "Deployment protection rules":
   - Check "Required reviewers"
   - Add `chipoto69` (the operator) as required reviewer
   - Set "Required reviewers" count to 1
6. Under "Deployment branches":
   - Choose "Protected branches only" (defaults to main + release branches)
7. (Optional) Add "Wait timer" if you want a delay before publish (default 0)
8. Click "Save protection rules"

## Verify

After save, the `release` environment shows:

- 1 required reviewer (chipoto69)
- Protected branches restriction

Trigger a test release by pushing a tag (e.g., `v0.0.0-h83-test`):

- The release.yml workflow runs, hits the `publish` job
- The job pauses with status "Waiting for approval"
- Operator (you) approves via the GitHub UI
- Job continues and drafts the release

After verification, delete the test tag + draft release.

## Status

Once the environment is configured and verified, H8.3 transitions from
`deferred-by-operator-choice` to `passed`. Update
`validation/F-B-hardening-WORKER-COMPLETE.md` to reflect the new status.

## Why this matters

Without the environment configured:

- The `publish` job in release.yml does NOT actually pause for approval
- Tags pushed to main auto-draft a release immediately
- No operator gate on production releases

With the environment configured:

- Every tag push waits for operator approval
- Required reviewers list is auditable in GitHub Settings
- Operator can scope to specific branches via "Deployment branches"
